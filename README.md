Overview

Stage202 introduces a Mini-PoC (Proof of Concept) that demonstrates how QSP (Quantum Security Protocol) claims are:

Explicitly declared

Bound to required CI jobs

Linked to concrete evidence artifacts

Automatically traceable via GitHub Actions

This stage establishes the credibility baseline for external review and PoC collaboration.

It is not a theoretical document.
It is a CI-verifiable security structure.

Mini-PoC Goals

The objective of Stage202 is to create a reproducible, closed-environment validation model:

Simulated VPS environment

Downgrade attack scenario

Drift injection scenario

Replay attack scenario

Interoperability smoke test

Each scenario produces:

Executable CI job

Evidence file under out/evidence/

Automatic inclusion in PoC report

Direct hyperlink to the CI job result

Architecture Principle

Stage202 enforces:

Fail-closed semantics

Explicit security claims (A2–A5)

CI-bound verification

Evidence path binding

No implicit trust assumptions

If a required CI job:

Fails

Is missing

Is not found in the run

→ The PoC report reflects that state.

Security claims cannot silently degrade.

Claims Structure

Declared in:

claims/claims.yaml

Each claim contains:

required_jobs

evidence_paths

Example structure:

claims:
  A2:
    required_jobs: ["attack_replay"]
    evidence_paths:
      - "out/evidence/attack_replay/result.txt"

tools/gen_poc_report.py

Security Design Philosophy

Stage202 does not claim:

Unconditional security

QKD guarantees

Formal proof completeness

Instead, it enforces:

Claim transparency

Execution-level verifiability

Traceable CI enforcement

Evidence visibility

This stage transforms abstract security claims into machine-linked artifacts.

Why Stage202 Matters

Before external PoC or institutional review (NICT, NTT, academic reviewers), a system must prove:

Claims are structured

Claims are testable

Claims are CI-enforced

Claims cannot silently regress

Stage202 is that threshold.

It is the minimum viable credibility layer.
